## 1. 世界观与底层逻辑公理（Lore & Absolute Axioms）

> **【系统指令】** 下列条目为 NSM 终端架构的**最高逻辑公理**。任何组件设计或数据流若与此背离，均视为**系统级致命缺陷（Fatal Error）**。

### 1.1 核心背景设定（Background Lore）

- **公理 I：零态军工集团（NullSpace Munitions，NSM）**
  - **实体性质：** 跨维度垄断型军工实体；立场为混沌中立。
  - **核心资产：** 微型虫洞生成引擎（Micro-Wormhole Engine）、绝对空间三维坐标网（Absolute Spatial Grid）。
  - **执行逻辑：** 不介入目标宇宙的政治或伦理判断；系统唯一触发条件为「等价交换」，即以**量子信用点（Quantum Credit，QC）**全额结算。

- **公理 II：目标接入节点（The Clientele / Operatives）**
  - **实体定义：** 具备多维跃迁能力、且合法持有 NSM 终端的主体（叙事代称：**跨界特工**）。
  - **系统认知：** NSM 终端不采集生物学特征或社会背景。在代码与持久化层，主体仅映射为挂载**神经钱包（Wallet）**的唯一标识符 `UUID`；即数字购买力载体，而非自然人档案。

- **公理 III：本源法则排斥律（Ontological Rejection）**
  - **物理常数壁垒：** 平行宇宙之间存在底层物理协议的互斥，不可混用。
  - **坍缩机制：** 任一脱离「原生宇宙（Native Universe）」的实体 SKU（武器），在暴露于异次元环境的瞬间，必然经历**亚原子级坍缩（Subatomic Collapse）**，物质归零为以太残渣。
  - **商品目录（Catalog）过滤约束：** 终端商品展示**必须且仅能**依据特工当前选定的「校准宇宙」施加等价于关系型查询 `WHERE` 的过滤条件；不得返回在当前宇宙法则下无法击发的 SKU。该约束属于**物理可行性门禁（硬性准入条件）**，而非可选的体验优化。

- **公理 IV：双向物流隔离协议（Logistics Dichotomy）**
  - 物质流转在叙事与实现层**严格二分**。禁止在变量命名、接口或文档中混用下列两套管线：
    - **【出向通道 · 军火交付】虫洞（Wormhole）：** 仅用于**向前线单向空投军火**的倾泻管线。**映射关系（禁止串线）：** 前端业务「虫洞倾泻军火」与后端 `wormhole_*` 协议栈一一对应。
    - **【入向通道 · 资产置换】黑洞（Black Hole）：** 仅用于**从指定坐标单向抽取实体物资并折算为 QC 余额**的管线。**映射关系（禁止串线）：** 前端业务「黑洞虹吸置换」与后端 `black_hole_*` 协议栈（核心持久化实体示例：`black_hole_intake_requests`，黑洞虹吸入账请求）一一对应。

- **公理 V：信息态亚空间基建（Subspace Data-Stream）**
  - **运行环境：** 为规避跨宇宙物理阻断，NSM 网络中枢与数据库不寄宿于任一物质宇宙，而锚定于纯数据流构成的**信息态亚空间（Subspace）**。
  - **交互实质：** 终端设备是以太神经节点；一切数据读写（CRUD）与身份鉴权（Auth），均为对亚空间总账本的**直接、实时**访问，不经由「本地宇宙」中转持久化。

### 1.2 多元宇宙设定（Multiverse Canon）

- **对象定义：** 「多元宇宙」在本项目中指**可并行存在、且彼此物理协议互斥**的平行世界集合；其叙事与商品侧表现须与 **1.1 公理 III**（原生宇宙、校准宇宙、目录过滤）及持久化层中的宇宙标识一致，不可在文档或接口中混用未声明的宇宙维度。

- **Nexus Lore Agency（设定生产管线）：**
  - **仓库位置：** `ai_agencies/nexus_lore_agency`。
  - **技术栈：** 基于 **CrewAI** 的多智能体（Multi-Agent）项目：以 YAML 配置代理与任务（`src/project/config/agents.yaml`、`tasks.yaml`），由 `crew` 编排执行，用于**撰写与修订各平行宇宙的设定稿件**（宏观物理与社会冲突、战争形态、武备与能量体系等），而非运行 NSM 终端业务逻辑。
  - **职责边界：** 专门承担「多元宇宙设定」的**生成与迭代**；产出经产品侧验收后，作为策划/叙事/目录文案的输入。**不**改写本章公理，**不**单独充当持久化层的唯一真源；落地文件以任务配置为准（例如默认 `report.md`）。
  - **代理分工（实现层摘要）：** 宏观概念解构与「概念外壳」提取；以及在武器商业与冲突视角下将概念重写为成体系的异世界设定报告（详见该目录内 `agents.yaml` / `tasks.yaml`）。

### 1.3 武器设定（Armament Specs）

- **对象定义：** 「武器设定」指每件 SKU 在叙事与展示侧所需的**客观文案真值**：名称、客观参数（尺寸、材质、能量源、原理与杀伤机制、缺陷与风险等），以及在给定世界观下的**设计合理性约束说明**。**不包含**面向文生图或任意图像生成模型的提示词（Prompt）、画幅或构图建议、镜头与光影类等**美术指令**；上述内容由 CAS **之外**的管线、工具或人工流程承担（若存在）。**面向成交与转化的营销话术、上架用语**与客观设定相区隔，由 **第 1.4 节** 所述管线负责。该类内容与 **1.1 公理 III**（SKU 与校准宇宙绑定、目录过滤）及持久化中的武器主数据对齐；生成稿须经验证后方可入库或上架。

- **上游依赖：** 宏观世界观输入优先来自 **第 1.2 节** 所述管线（任务语境中即「世界观设定局」所供背景）；武器管线在**设定前提已给定**的前提下工作，**不**反向改写多元宇宙公理叙事。

- **Conceptual Armory Studio（CAS，多元宇宙武器设定管线）：**
  - **仓库位置：** `ai_agencies/conceptual_armory_studio`。
  - **技术栈：** 基于 **CrewAI** 的多智能体项目：以 `src/project/config/agents.yaml`、`tasks.yaml` 配置，由 `crew` 编排执行；**不**承担 NSM 终端运行时或交易闭环逻辑。
  - **PRD 职责（权威）：** **仅为各多元宇宙设计武器**——在给定世界观与约束下，产出可按宇宙维度归档的武器条目及其客观描述（杀伤机理、材质与能量、尺寸量级、风险与缺陷等）。**明确不包含：** Prompt 撰写、推荐画幅或画面比例、文生图/图生图指令、视觉资产管理。**交付形态以实现为准**（如 Markdown 清单或内部汇报稿）；本章仅以「武器设定文案」为契约对象，**不要求** CAS 产出任何图像或可执行的绘图 Prompt。
  - **职责边界：** 专门承担「多元宇宙武器设定」的**生成与迭代**；**不**改写本章公理，**不**单独充当持久化层的唯一真源；数值平衡、定价、库存与交付协议以产品与后端 schema 为准。代理角色与任务拆分属于实现细节；与上述 PRD 契约冲突时，**以 MASTER_SPEC 为准并回溯修订实现**。

### 1.4 商品营销文案（Commerce / Merchandising Copy）

- **对象定义：** 「营销文案」指附着于每件武器 SKU、**以促成浏览与购买转化为目标**的商业化表述：包括但不限于面向终端或电商渠道的商品标题、短描述、卖点提炼、促销话术等。**不得**与 **第 1.3 节** 的客观设定相混淆：营销层允许夸张与风格化，但须在验收规则中与物理可行性、校准宇宙及价格策略进行核对，避免与公理或主数据冲突。

- **上游依赖：** 以 **第 1.3 节（CAS）** 产出的武器客观描述与参数为输入做「数据富化」；**不**替代 CAS 进行武器机理设计，**不**改写 **第 1.2 节** 的宇宙级设定。

- **Galactic Sales Agency（GSA，星际销售署 / 商品营销管线）：**
  - **仓库位置：** `ai_agencies/galactic_sales_agency`。
  - **技术栈：** 基于 **CrewAI** 的多智能体项目：以 `src/project/config/agents.yaml`、`tasks.yaml` 配置，由 `crew` 编排执行；**不**承担 NSM 终端运行时、支付结算或订单履约逻辑。
  - **PRD 职责（权威）：** **为每一把武器撰写营销文案**——在既定武器事实前提下，按 SKU（及所属宇宙语境）生成面向上架与导购的商业表述；字段粒度与排版以实现为准（例如任务中可拆分为商业花名、销售短评、核心卖点等），**交付形态以实现为准**（如 Markdown 商品目录便于后续清洗入库）。
