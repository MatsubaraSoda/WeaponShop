## 1. 世界观与底层逻辑公理（Lore & Absolute Axioms）

> **【系统指令】** 下列条目为 NSM 终端架构的**最高逻辑公理**。任何组件设计或数据流若与此背离，均视为**系统级致命缺陷（Fatal Error）**。

### 1.1 核心背景设定（Background Lore）

- **公理 I：零态军工集团（NullSpace Munitions，NSM）**
  - **实体性质：** 跨维度垄断型军工实体；立场为混沌中立。
  - **核心资产：** 微型虫洞生成引擎（Micro-Wormhole Engine）、绝对空间三维坐标网（Absolute Spatial Grid）。
  - **执行逻辑：** 不介入目标宇宙的政治或伦理判断；系统唯一触发条件为「等价交换」，即以**量子信用点（Quantum Credit，QC）**全额结算。

- **公理 II：目标接入节点（The Clientele / Operatives）**
  - **实体定义：** 具备多维跃迁能力、且合法持有 NSM 终端的主体（叙事代称：**跨界特工**）。
  - **系统认知：** NSM 终端不采集生物学特征或社会背景；持久化层不承认「自然人档案」，只承认可记账的购买力。
    - **身份即钱包（Identity–Wallet Fusion）：** **`wallets` 表主键 `id` 与鉴权主体 ID（如 Supabase `auth.users.id`）必须为同一 UUID**——不在此处另造独立 `wallet_id`、不设「用户 ID / 钱包 ID」二分；系统**不**支持一用户多钱包（无多子账户、无多币种子账户；与 **公理 I** 单轨 QC 一致）。链入终端即激活**资产锚点**：在信息态亚空间中，「你是谁」退居其次，**余额与流水所锚定的 UUID** 即主体在系统内的可操作身份。
    - **实现侧（如 Supabase）：** `auth.uid()` 可直接作为 `wallets.id` 参与查询与 RLS，无需先解析「用户表 → 钱包表」二次跳转。

- **公理 III：本源法则排斥律（Ontological Rejection）**
  - **物理常数壁垒：** 平行宇宙之间存在底层物理协议的互斥，不可混用。
  - **坍缩机制：** 任一脱离「原生宇宙（Native Universe）」的实体 SKU（武器），在暴露于异次元环境的瞬间，必然经历**亚原子级坍缩（Subatomic Collapse）**，物质归零为以太残渣。
  - **「可击发」与「以太残渣」之逻辑一致：** 在本世界观设定中，**不存在**与 **坍缩机制** 并列的第二套独立物理禁令。终端与文档所称「在当前校准 / 目标语境下**不可击发**」，与「若将该 SKU 置于该语境并触发不相容法则，则物质将按上款归零为**以太残渣**」刻画**同一边界**：**未残渣**即可操作，**残渣**即不可击发——二者互为表里，仅叙事侧重不同（一为终端门禁用语，一为坍缩后果）。
  - **商品目录（Catalog）过滤约束：** 终端商品展示**必须且仅能**依据当前**校准宇宙**（见文首术语：**专指终端浏览上下文**）施加等价于关系型查询 `WHERE` 的过滤条件；不得返回在该浏览上下文对应物理法则下无法击发的 SKU。该约束属于**物理可行性门禁（硬性准入条件）**，而非可选的体验优化。

- **公理 IV：双向物流隔离协议（Logistics Dichotomy）**
  - 物质流转在叙事与实现层**严格二分**。**命名约定：** **出向**全称 **「虫洞倾泻协议」**，简称 **「虫洞协议」**（实现前缀 `wormhole_*`）；**入向**全称 **「黑洞虹吸协议」**，简称 **「黑洞协议」**（实现前缀 `black_hole_*`）。禁止在变量命名、接口或文档中混用两套管线。
    - **【出向 · 军火交付】虫洞倾泻协议（简称虫洞协议，Wormhole）：** 仅用于**向前线单向空投军火**的倾泻管线。**映射关系（禁止串线）：** 前端业务与叙事可使用简称「虫洞协议」或与全称一致之表述，并与后端 `wormhole_*` 协议栈一一对应。
    - **虫洞协议 · 运输风险（叙事 / 条款）：** 买方支付价金并与 NSM 达成合意后，出向管线占用即视为履行。**若叙事上**发生虫洞扰动、坐标漂移、亚空间湍流等导致物资**未抵达或未完好抵达**，NSM 将该类情形归入**类自然灾害 / 不可抗力**：**不设退款、不提供售后赔付**；亚空间总账侧以**结算事务成功**为准（详见 **§2.4**）。
    - **虫洞协议 · 终端约束（实现）：** 为降低全栈实现复杂度，**虫洞倾泻协议**不提供预约或延迟执行时间选择；用户**提交虫洞协议即视为即时发起运输**。持久化层不得把「计划执行时间」作为协议必填字段或前置条件。
    - **【入向 · 资产置换】黑洞虹吸协议（简称黑洞协议，Black Hole）：** 仅用于**从指定坐标单向抽取实体物资并折算为 QC 余额**的管线。终端侧提交**战术代号**（持久化 **`protocol_name`**，与虫洞协议命名语义对齐）、物质所在**目标宇宙**与**绝对坐标**（`coord_x` / `coord_y` / `coord_z`）；**不得**由特工在协议中**自报**「应值多少 QC」——**入账数额由 NSM 在亚空间内根据实收物质与统一定价规则计算后写入**（与 **§2.3 定价权**、持久化 `credited_amount_qc` 一致）。**映射关系（禁止串线）：** 前端业务「黑洞虹吸置换」与后端 `black_hole_*` 协议栈（核心持久化实体示例：`black_hole_protocols`）一一对应。
    - **物流协议状态（实现）：** `black_hole_protocols` 与 `wormhole_protocols` 的 **status** 均为同一三态枚举：**`DRAFT`**（编制未提交）、**`PENDING`**（已提交、处理中）、**`COMPLETED`**（终态成功）。**不**引入 FAILED / ABORTED / 协议多段中间态等额外枚举，以降低全栈分支。**持久化层「失败」仅表现为**：数据库**事务未能成功提交**时的回滚与接口错误（见 **§2.5**）；**不**将叙事侧虫洞运输事故建模为协议 FAILED 行。**虫洞**价金结算与 `COMPLETED` 触发以 **§2.4**（模拟事务闭环）为准。

- **公理 V：信息态亚空间基建（Subspace Data-Stream）**
  - **运行环境：** 为规避跨宇宙物理阻断，NSM 网络中枢与数据库不寄宿于任一物质宇宙，而锚定于纯数据流构成的**信息态亚空间（Subspace）**。
  - **交互实质：** 终端设备是以太神经节点；一切数据读写（CRUD）与身份鉴权（Auth），均为对亚空间总账本的**直接、实时**访问，不经由「本地宇宙」中转持久化。

### 1.3 武器设定（Armament Specs）

  - **对象定义：** 「武器设定」指每件 SKU 在叙事与展示侧所需的**客观文案真值**。**实体公设：** 纳入目录的武器 SKU 均视为**具实体**（占有空间、由物质构成），以便统一最小字段而不无限细分品类。**不包含**面向文生图或任意图像生成模型的提示词（Prompt）、画幅或构图建议、镜头与光影类等**美术指令**；上述内容由 CAS **之外**的管线、工具或人工流程承担（若存在）。**面向成交与转化的营销话术、上架用语**与客观设定相区隔，由 **第 1.4 节** 所述管线负责。该类内容与 **1.1 公理 III**（SKU 与**原生宇宙**、目录随**校准宇宙/浏览上下文**过滤）及持久化中的武器主数据对齐；生成稿须经验证后方可入库或上架。
  - **完整客观描述（脚手架：`weapon_armament_spec.description`）：** **CAS 视角**下该 SKU 的**完整武器客观设定正文**（可供列表 / 详情 / 检索展示）；**五元**为同一条目之**结构化分项**（便于校验、虫洞锁价与目录逻辑），与 `description` 互补而非要求逐字重复。**不得**与 **§1.4** 营销 **`short_description`** 混用（后者为转化话术，允许风格化）。**宇宙**侧完整客观设定见脚手架 **`universes.description`**（与 `universes.name` 区分）。
  - **客观参数 · 通用五元（脚手架：`weapon_armament_spec` 文本列）：** 不同宇宙的机理差异很大，以下五项为**跨宇宙共通语义槽**（叙述形态可为自然语言短段落）：
    1. **名称（怎么称呼）** —— 客观侧称谓，可与导购商品名（**§1.4**）不同。
    2. **尺寸** —— 基于实体公设的尺度描述（外形占位 / 量级）；更细的宇宙特异度量可放进 **`extended_spec`**。
    3. **原料** —— **制成**该实体过程中消耗 / 转化的物质来源（武器由原料构成的叙事闭合）。
    4. **使用依赖** —— **使用阶段**所依赖的介质或能源（**不是**「必有化学燃料」：可为弹药、工质、电池、封装药剂；亦可显式写**人力驱动**、**无化学消耗**等；统一避免与「实体公设」矛盾的「真空魔法」表述，除非写入 **`extended_spec`** 并由验收认可）。
    5. **原理** —— 作用机理：为何能实现杀伤 / 自卫 / 压制（与 **公理 III** 物理可行性叙事对齐）。
  - **客观参数 · 宇宙特异（脚手架：`extended_spec jsonb`）：** 非全局通用的键置于 **PostgreSQL `jsonb`**（如虫族生化 SKU 的「活性基因」、机械宇宙没有的字段）。**禁止**在 JSON 中重复存放 **`description` 与五元**已表达的语义（避免双写）；扩展键仅承载**无法落入上述字段**或**仅某宇宙生效**的属性。
  - **可选附录字段：** **固有缺陷与风险**、**设计合理性约束说明**等仍以可空文本列或迁移入 **`extended_spec`** 之一为准（以实现与 CAS 产出形态为准），不在**完整 description 与五元**内强制，以免表单膨胀。

- **上游依赖：** 宏观世界观输入优先来自 **第 1.2 节** 所述管线（任务语境中即「世界观设定局」所供背景）；武器管线在**设定前提已给定**的前提下工作，**不**反向改写多元宇宙公理叙事。

- **Conceptual Armory Studio（CAS，多元宇宙武器设定管线）：**
  - **仓库位置：** `ai_agencies/conceptual_armory_studio`。
  - **技术栈：** 基于 **CrewAI** 的多智能体项目：以 `src/project/config/agents.yaml`、`tasks.yaml` 配置，由 `crew` 编排执行；**不**承担 NSM 终端运行时或交易闭环逻辑。
  - **PRD 职责（权威）：** **仅为各多元宇宙设计武器**——在给定世界观与约束下，产出可按宇宙维度归档的武器条目及其**完整客观 `description` + 五元结构化描述 + 可选缺陷/约束**，并将宇宙特异属性置于 **`extended_spec`** 可序列化结构中。**明确不包含：** Prompt 撰写、推荐画幅或画面比例、文生图/图生图指令、视觉资产管理。**交付形态以实现为准**（如 Markdown 清单或内部汇报稿）；本章仅以「武器设定文案」为契约对象，**不要求** CAS 产出任何图像或可执行的绘图 Prompt。
  - **职责边界：** 专门承担「多元宇宙武器设定」的**生成与迭代**；**不**改写本章公理。武器数值平衡、标价口径、库存与交付是否在终端采纳，**以本文（含第 2 章）及已定产品规则为最高准则**；`docs/schema`、迁移与运行时库表仅是落地脚手架，**须与 PRD 对齐**，冲突时**改脚手架与流水线**，不反向改公理或歪曲本文已采纳规则。代理角色与任务拆分属于实现细节；若管线产出与 PRD 不符，**以 MASTER_SPEC 为准并回溯修订实现**。

## 2. 终端操作协议与业务持久化（Terminal Operational Protocols & Persistence）

**权威顺序（持久化）：** **第一级** — **`docs/MASTER_SPEC.md`（本文）**：字段语义、状态机、门禁与业务不变量为唯一来源。**第二级** — **`docs/schema/all.dbml`**：全库合并 ER / 表结构与 FK / 索引之**规范性快照**；实现迁移、dbdiagram 导入与代码模型以与 **第二级** 对齐为目标。**脚手架（域拆分）** — **`docs/schema/nsm_universe_weapons.dbml`、`nsm_wormhole.dbml`、`nsm_black_hole.dbml`、`nsm_transaction.dbml`**：起草第一、二级规范时的**分域导读**，便于分文件讨论；**不与 `all.dbml` 同级**，若与 **第一级 / 第二级** 冲突，**修订域文件并回溯同步 `all.dbml`**，再以迁移与代码对齐 **本文**。不得反向以脚手架架空 PRD。

*   **Supabase / PostgreSQL：** NSM 终端业务表部署于 Supabase 托管 Postgres，默认 schema **`public`**。**第二级** `all.dbml` 已用限定表名 **`public.<table>`**（如 **`public.wormhole_protocols`**），与 `CREATE TABLE public.*`、查询限定名及 SQL Editor 一致。**`auth`** schema（例如 **`auth.users`**）为 Supabase Auth 托管对象，**不在** NSM 业务 DBML 中建表定义，仅在注释、`auth.uid()` / RLS 中引用。

*   **协议归属列（`wallet_id`）：** **`wormhole_protocols`、`black_hole_protocols`** 使用 **`wallet_id` → `public.wallets.id`**（**不**直接 FK **`auth.users`**）。终端叙事里发起协议的主体是**挂载 QC 的钱包锚点**，而非抽象「注册用户」；**`public`** 业务域与 **`auth`** 仅通过 **`wallets.id = auth.users.id`** 一条纽带衔接，便于迁移与解耦。RLS / 查询可统一 **`wallet_id = auth.uid()`**（UUID 与钱包主键同值）。详见 **第二级** **`docs/schema/all.dbml`**。

### 2.1 亚空间链入与身份锚定（Registration & Authentication）
*   **链入机制：** 跨界特工通过终端发起「亚空间链入」请求（映射为 Supabase Auth 等服务）。
*   **身份映射与钱包（公理 II · 身份即钱包）：** 成功鉴权后，`auth.users.id`（UUID）即亚空间内唯一**身份锚点**；**同一份 UUID 必须作为 `wallets.id` 写入**，不得在 `wallets` 上再设 `user_id` 等列重复指向 `auth.users`。
*   **钱包行初始化：** 首链入时**原子化**插入 `wallets` 一行：`id = auth.users.id`，`balance_qc` 初始为 `0`（或产品约定初值），`status` 默认 `ACTIVE`。
    *   **字段：** `balance_qc` 为 QC 余额唯一真源。
    *   **禁令：** 严禁绕过 `wallet_transactions` 直接改写 `balance_qc`；严禁人为制造同一 `auth.users.id` 下多行 `wallets`。

### 2.2 资产置换账本（Neural Ledger & Transactions）
*   **QC 数值约定（整数、非小数类型）：** 全部 QC 金额（`balance_qc`、`amount_qc`、`list_price_qc`、`*_qc` 标价与快照等）**均以整数持久化**，语义为 **QC 最小货币单位**，与脚手架 **`bigint`** 一致。**不**使用 **`DECIMAL` / `NUMERIC`**（会计小数）作为 MVP 持久化方案；**不**使用 **`float` / `double`** 表示 QC，避免舍入误差。**不**为「会计 Decimal」单独引入类型层——若未来需更细粒度，通过**约定更小整数单位**（等价放大倍数）解决，而非库内小数列。
*   **不可变流水（`wallet_transactions`）：** 终端的一切价值波动必须记录于不可变账本。
*   **交易类型（双向隔离）：** 为符合 **公理 IV**，账本类型**仅限两种**且严禁串线：
    *   **`BLACK_HOLE_INTAKE`：** 对应 **黑洞虹吸协议（黑洞协议）** 入账，必须关联 `related_black_hole_protocol_id`。
    *   **`WORMHOLE_CONSUME`：** 对应 **虫洞倾泻协议（虫洞协议）** 扣款，必须关联 `related_wormhole_protocol_id`。
*   **记账法则：** `amount_qc` 正值表示黑洞协议侧入账，负值表示虫洞协议侧支出。系统不提供退款（Refund）逻辑。**区分两层：**（1）**事务未成功提交**（门禁失败、余额不足、校验错误等）——整笔业务回滚，**不写**扣款与协议前进；（2）**事务已成功提交**——`WORMHOLE_CONSUME` 等流水与余额变动**生效**，**不因**叙事侧「虫洞传输失败」类事由冲正（见 **§2.4、§2.5**）。

### 2.3 黑洞虹吸协议（简称黑洞协议）— 入向持久化
*   **协议定义：** **黑洞虹吸协议**管入向：特工在终端提交**战术代号**（`protocol_name`，与虫洞侧命名对齐）、物质所在**目标宇宙**与**绝对坐标**（`coord_x` / `coord_y` / `coord_z`）；**不由用户填报拟入账 QC**。NSM 自远端完成抽取后，在亚空间内对实收物质**估值**，并将**最终入账 QC 数额**写入协议行（`credited_amount_qc`）与 **`BLACK_HOLE_INTAKE` 流水**。
*   **定价权与叙事（Lore）：** NSM 对入向物资保有**唯一定价权**（「集团侧裁定」）。虹吸物质穿过事件视界后先经**亚空间验资帷幕（Subspace Valuation Veil）**：在军工采购中枢与跨宇宙行情管线中完成**理化扫描与等价折算**，终端与特工**不参予**报价磋商；**等价交换**在此表现为「代号与坐标锚定抽吸 → **入账瞬间**数额由 NSM 写入总账本」。叙事上可简称：**你填代号与坐标，QC 由集团点数**（代号用于终端侧区分编制单，**不**参与报价）。
*   **创建来源（MVP）：** **`black_hole_protocols`** **仅**通过终端 **§3.5 `/black-hole`** 创建表单（或等价服务端接口）由用户**显式新建**。**不**在注册、首链入或 **§2.1** 钱包初始化流程中自动插入占位行。**MVP 不设**黑洞 / 虫洞**协议手续费**（见 **§4** 章首说明）。
*   **流转状态机（`black_hole_protocols`）：**
    *   **`DRAFT`**：编制 **`protocol_name`**、`target_universe_id` 与 `coord_x` / `coord_y` / `coord_z`，未提交；**无**用户填报之「拟入账 QC」字段。
    *   **`PENDING`**：已提交，亚空间执行抽取与验资；`credited_amount_qc` 在 **`COMPLETED` 前**保持**空**（`NULL`）。
    *   **`COMPLETED`**：**NSM 评估完成**，写入 `credited_amount_qc`，并据此注入钱包、记 **`BLACK_HOLE_INTAKE`**；以本状态写入时的 `updated_at` 作为入账完成时刻。
*   **原子性约束：** 仅当 **黑洞协议** 行变为 `COMPLETED` 时，方可触发钱包余额更新及对应 **`BLACK_HOLE_INTAKE` 流水**；流水 `amount_qc`（正值）须与 **`credited_amount_qc`** 一致。
*   **MVP · 无明细子表与模拟估值（库表见 **第二级** `docs/schema/all.dbml`）：** **不**设黑洞协议行项目表；入向只读壳层为 **`black_hole_protocols`**。协议在 **`COMPLETED`** 时，由 **NSM 侧**（服务端事务）一次写入 **`credited_amount_qc`** 并增加 **`wallets.balance_qc`**、记 **`BLACK_HOLE_INTAKE`**。**估值裁定（实现占位）：** 在 **闭区间 \[1000, 1000000\]** 内取 **均匀随机整数 QC**（含端点），写入 **`credited_amount_qc`** 与同笔流水 **`amount_qc`**——象征「唯一定价权 / 单方面裁定」，**非**终版理化扫描管线。**后续版本**可替换为真实估值逻辑；叙事边界仍以 **不由用户自报入账数额** 为准（见上条协议定义）。
*   **战术代号唯一（与虫洞对称）：** **同一 `wallet_id` 名下 `protocol_name` 不得重复**（创建前校验；与数据库唯一约束一致，见 **第二级** **`docs/schema/all.dbml`**）。
*   **DRAFT 编制规则（与 §2.4 虫洞对称）：** 黑洞协议行一旦**创建成功**（进入 `DRAFT`），**不得**再修改 `target_universe_id`；物质所在宇宙若需变更，须**新建**一条黑洞协议。`protocol_name`、`coord_x` / `coord_y` / `coord_z` 仍可在 `DRAFT` 内调整。终端 **§3.5** 以 **Carousel** 锚定创建时之 `target_universe_id`（与 **§3.4** 虫洞页同型）；**Sheet** 内 **`target_universe_id`** **只读展示**，避免与 Grid 过滤语义冲突。

### 2.4 虫洞倾泻协议（简称虫洞协议）— 出向持久化
*   **协议构成：** **虫洞倾泻协议**由 `wormhole_protocols` 与 `wormhole_items` 组成，出向单向倾泻；**同一协议内同一 `weapon_id` 仅允许一行**（只改 **`quantity`**、不允许两条重复 `weapon_id`），见 **第二级** **`docs/schema/all.dbml`** 之 **唯一约束**——应用侧凡「再加同款武器」必须合并数量 **UPSERT**，**不得**依赖插入第二行。
*   **交付约束（即时性）：** 依 **公理 IV · 虫洞协议 · 终端约束**，**虫洞协议提交即即时运输**。
    *   **去字段化：** 持久化层不得增加「计划执行时间」等预约类列。
*   **行级锁价 `price_snapshot_qc`：** **`DRAFT` 下为 `NULL`**；协议进入 **`PENDING`** 时，自各行的 `weapons.list_price_qc` 写入 **`wormhole_items.price_snapshot_qc`**，其后 **`PENDING` / `COMPLETED`** 均以此快照计价，**不再**随目录价变动。
*   **协议总价 `total_cost_qc`（`wormhole_protocols`，`NOT NULL`，默认 `0`）：**
    *   **`DRAFT`：** 无行时为 **`0`**；有行时由**服务端**按当前目录价实时汇总：各行 **`weapons.list_price_qc × wormhole_items.quantity`** 之和（**`price_snapshot_qc` 为 `NULL`**，单价取目录）。展示与门禁均以该实时值为准。
    *   **`PENDING` / `COMPLETED`：** **`total_cost_qc` 为进入 `PENDING` 当刻的快照总价**（与各 **`price_snapshot_qc × quantity`** 之和一致），**不再**按目录价重算。
*   **状态流转：** `DRAFT` → `PENDING`（扣款并锁价）→ `COMPLETED`（出向业务闭环，见下条）。
*   **模拟项目 · `PENDING` → `COMPLETED`（与 Supabase 一致）：** 本工程为**模拟终端**；**MVP** 下在 **Supabase** 侧**同一数据库事务**内完成：锁价、扣减 `wallets.balance_qc`、写入 `WORMHOLE_CONSUME`、将 `wormhole_protocols.status` 自 `PENDING` 置为 **`COMPLETED`** 等**全部成功并提交**后，即视为出向协议**终态闭环**——**不**再等待真实世界「货已落地」之外部事件。实现上可在该事务末段**直接**将状态置为 `COMPLETED`。
*   **虫洞运输风险与 Lore（类自然灾害、不退款）：** 叙事上可存在**出向未抵达**、管道路径**灭失**等「运输失败」；对 NSM 而言，买卖合意与价金在**亚空间总账事务成功**时即已完成**结算侧**的等价交换。此类运输侧事故在条款上**类同自然灾害 / 不可抗力**：**无售后、不赔付、不退还**已扣 **QC**；**不得**因叙事失败回滚已提交之 **`WORMHOLE_CONSUME`** 与 **`COMPLETED`** 行。产品陈述可简写为：**一个愿买、一个愿卖，钱货在总账上两清后，管线上出事不算「未成交」**。
*   **DRAFT 编制规则（减轻全栈分支）：** 虫洞协议行一旦**创建成功**（进入 `DRAFT`），**不得**再修改 `target_universe_id`；投送目的地宇宙若需变更，须**新建**一条虫洞协议，不得在同一行上「改目的地」。`coord_x` / `coord_y` / `coord_z` 及未锁价前的明细等**编制中**字段，仍可在 `DRAFT` 内调整。
*   **MVP 与审计：** **Chingu Solo Project Tier 3** 范围下，**不**要求虫洞行级「修改审计」时间戳等字段；以 **`created_at` / 协议级 `updated_at`** 等产品已有列为准，避免需求膨胀。

### 2.5 异常处理与逻辑门禁（Error Handling & Guardrails）
*   **无失败终态（持久化枚举）：** 遵循 **公理 IV**，协议行**不**使用 FAILED / ABORTED 等终态枚举。**「失败」分两景：**（**A**）**事务未能提交**——余额不足、门禁不通过、校验错误等：整笔 **回滚**，协议停留在 `DRAFT` 或不产生前进；用户收到错误码。（**B**）**事务已提交**——虫洞价金已扣、`COMPLETED` 已写入（见 **§2.4**）：叙事侧若演绎「运输灭失」，**不**改库为失败态、**不退款**，而归入 **§2.4** 之不可抗力叙事。
*   **物理准入门禁（公理 III）— 虫洞倾泻：** 创建或提交 **虫洞倾泻协议** 时，须满足：**协议** `target_universe_id` 与**各行** `wormhole_items` 所引用武器的 `weapons.native_universe_id`**一致**（同协议内 SKU 同属该投送目的地宇宙）；且每个 `weapon_id` 在该宇宙之物理可行性下**允许击发**（见公理 III）。**不**将「特工当下所处物质宇宙」作为硬编码约束——终端可切换**校准宇宙**（见文首术语：终端浏览上下文）以查看 **B / C / D…** 等任意目录并代他人备货；与 **§4.1** 用户故事一致。否则拒绝建单或拒绝进入 `PENDING`。
*   **虫洞 · `DRAFT` → `PENDING` 门禁分层（与 §4.4）：** 因 **§2.4** 当前流程下 **`PENDING` 一经达成即在同一事务内直达 `COMPLETED`**（暴力闭环，亚空间黑市不做「挂单反悔」），**提交前**门禁必须做足。**必要条件：**（**1**）**余额充足**——**必须由 Supabase 侧**（服务端事务 / RPC / 约束）校验，**列为第一要素**；（**2**）**宇宙一致**——`wormhole_protocols.target_universe_id` 与各行武器 **`native_universe_id`** 一致：**默认可由前端**在提交前完成核对（黑市终端「买家自负对齐」叙事）；若 **Supabase** 端以触发器、RPC、`CHECK`、关联查询等**低成本即可实现**，**建议同步加上**服务端校验，防篡改与脏提交。

## 3. NSM 终端 UI/UX 设计规范（Terminal UI/UX）

**权威顺序：** **以本文（MASTER_SPEC）及第 1、2 章为最高准则**；持久化以 **第二级 `docs/schema/all.dbml`** 为库表真源，**域拆分 `nsm_*.dbml`** 仅为脚手架 / 导读；若 UI 本章已写明字段语义而 **`all.dbml`** 暂缺列名，**以本文为准并回溯修订 `all.dbml`（及必要时域脚手架）**。本章目前规定认证路由、全局导航、首页 **§3.3** 三区块、**§3.4** 虫洞协议管理页 **`/wormhole`**、**§3.5** 黑洞协议管理页 **`/black-hole`**，以及 **§3.6** QC 钱包页 **`/wallet`**；其余细节可后续增补。

### 3.1 认证区域（`/auth/*`）

*   **路由：** 登录 **`/auth/login`**；注册 **`/auth/register`**。
*   **版式：** 页面**极为干净**——视口内**仅**居中一枚 **Card**（表单置于 Card 内）；**不得**挂载全局 **Nav**，**不得**展示页脚信息或其它站点框架元素。

### 3.2 业务壳层与全局导航（非 `/auth`）

*   **适用范围：** 除 **`/auth/*`** 外，业务路由共用**同一套壳**：顶部 **Nav + 下方主内容区**。
*   **Nav 三区布局（左 · 中 · 右）：**
    *   **左：** **Logo**。
    *   **右：** **语言切换**、**深浅色（主题）切换**、当前会话用户 **UUID**（与 **`auth.users.id` / 公理 II** 一致，仅展示标识，具体样式由实现定）。
    *   **中：** 若干 **Dropdown** 的 **trigger**。**MVP：** 各下拉 **无菜单子项**（`DropdownMenuItem` 等**暂空**），仅 **trigger** 可点；**已确定四项：**
        *   **「NSM 终端」** → 导航至首页 **`/`**。
        *   **「虫洞倾泻协议」**（出向 `wormhole_*`）→ 进入 **`/wormhole/*`** 命名空间（入口 **`/wormhole`**；签订动作见 **§3.3 / §3.4** 各 **`Sheet`**，**无**独立签订路由）。
        *   **「黑洞虹吸协议」**（入向 `black_hole_*`）→ **`/black-hole`**（见 **§3.5**；持久化前缀仍为 `black_hole_*`）。
        *   **「QC 钱包」**（`wallets` / **§2.2** 账本）→ **`/wallet`**（见 **§3.6**）。

### 3.3 首页 `/`

*   **整体：** Nav 下方主区域为**垂直布局**（自上而下区块排列）。
*   **第一区块：校准宇宙 — Carousel**
    *   使用 **Carousel** 组件承载「校准宇宙」切换；滑动 / 切换条目即更新当前**校准上下文**（文首术语），并驱动下游军火列表数据源。**实现上**应将「校准宇宙」抽取为**可复用的 React 组件**（**§3.4 `/wormhole`**、**§3.5 `/black-hole`** 与首页首区块**共用**同一实现），避免多页 Carousel 行为漂移。
    *   **普通 CarouselItem（对应某一平行宇宙）：** 视觉为一张 **16:9 横版配图**（持久化字段 **`universes.cover_image_url`**，库表见 **第二级** **`docs/schema/all.dbml`**）；文案区展示 **`code`、`name`、`description`** 三项。
    *   **欢迎 CarouselItem（额外一项）：** 使用一张**欢迎主题配图** + **欢迎文案**（文案内容后续定稿）。当 Carousel **选中该欢迎项**时，军火列表进入**全宇宙聚合模式**：展示**所有宇宙**的武器条目（**不按**单一 `native_universe_id` 过滤）；用于「一览全部挂载 SKU」；与选中某一宇宙条目时的**单宇宙校准过滤**并列存在。
*   **排序建议：** 除「欢迎」项的位置由产品设计固定外，其余宇宙条目建议按 **`universes.code` 字符串升序**排列（与 **§1.2** 一致）。

*   **第二区块：虫洞倾泻协议 — Select（shadcn）**

    *   **组件：** 采用 **shadcn/ui `Select`**（或与其交互模型一致的封装）。本区块**仅承担「选定当前会话要使用的虫洞协议」**，且**仅限 `DRAFT`**；**不包含**新建、**删除整条协议**、改协议级坐标、或变更协议壳层字段——此类**协议管理**见 **§3.4** **`/wormhole`**。**允许**在本区块配套的「购物篮」Sheet 内**增减行项目**（`wormhole_items` 级），本质为编制购物车，**不等同于**删除协议实体。
    *   **数据范围（选项 `items`）：** 查询当前登录用户的 **`wormhole_protocols`**（**`wallet_id` = `auth.uid()`**，即当前会话钱包主键），且**必须**附加过滤：**`status = 'DRAFT'`**——首页第二区块**仅用于编制中**的出向协议；**`PENDING` / `COMPLETED`** 的协议**不出现在此 Select**（查阅与履约改走 **`/wormhole`**（§3.4））。
        *   当第一区块 Carousel **选中某一平行宇宙**（映射到 **`universes.id`**）时：在上述 **`DRAFT` 约束**下，**仅列出** `target_universe_id` **等于**该校准宇宙 id 的虫洞协议。
        *   当 Carousel **选中「欢迎」项**时：在上述 **`DRAFT` 约束**下，**列出该用户名下全部**（不限 `target_universe_id`）**编制中**虫洞协议。
        *   若当前过滤条件下**不存在**任何虫洞协议：**`items` 为空**，Select 无可选项（见下文空态文案）。
    *   **展示建议：** 选项文案优先 **`protocol_name`（战术代号）**；**可不展示** `status` 徽标（选项恒为 `DRAFT`），若需统一组件样式保留「编制中」弱标签亦可。
    *   **版面与层级：** 置于 Carousel **下方**；含**区块标题 + 辅助说明 + Select +「查看购物篮」触发按钮**（见下）。

    *   **购物篮视图：`Sheet`（shadcn/ui）**
        *   **触发：** 在 Select 旁（或同一工具行）提供次要按钮，文案建议：**「查看出向清单」** / **「查看协议内武器」**；点击后以 **shadcn `Sheet`** 自**视口右侧**滑入（**`side: 'right'`**，与常见电商购物车一致；**不**改为自左侧打开，以免与本条「面板内箭头」心智冲突）。
        *   **面板内顶栏（左上）：** 在 **Sheet 内容区最上方**（或 **SheetHeader** 内）靠**左**布置一条**窄控件**，内含两枚**图标按钮**（建议 **`ChevronLeft` / ChevronRight`** 或 **`ArrowLeft` / ArrowRight**），使用户**不必返回第二区块**即可操作 Sheet：
            *   **收起（关闭）：** 其中一枚箭头映射为**关闭 Sheet**（与 **shadcn `SheetClose`**、点击遮罩外、`Esc` **同一语义**：`open` → `false`）。**建议映射：** **向左箭头** = 收起（侧栏自右侧滑出，箭头朝向主内容区，表示「推回」主界面）。
            *   **展开（可选）：** 另一枚映射为 **Sheet 加宽态**（例如 `max-w-sm` → `max-w-2xl` 或全屏占比阶梯切换），便于浏览长清单；若 **MVP** 仅固定单一宽度，可**暂不接线**该按钮（隐藏或禁用），但**版式上预留**位置以免日后改版抖动。
            *   **实现要点：** 使用 **`open` / `onOpenChange` 受控状态**（或与 shadcn 推荐写法等价），顶栏按钮与第二区块触发按钮**共享同一布尔状态**——故「打开」仍可从第二区块点，**关闭**可在面板内、遮罩、`Esc` 任意一处完成。
        *   **Sheet 内禁止：** **不得**在本 Sheet 内提供「新建协议 / 删除整条协议 / 重命名协议壳」等入口；此类操作在 **`/wormhole`**（§3.4）完成。
        *   **同 SKU 仅一行（必选数据语义）：** `wormhole_items` 在库中为 **`(wormhole_protocol_id, weapon_id)` 唯一**（见 **第二级** **`docs/schema/all.dbml`**，与 **§2.4**）。因此从首页第三区块（或其它入口）向当前选定 **`DRAFT`** 协议**再次添加同一 `weapon_id`** 时，**禁止**再插入第二行，必须 **`quantity` 合并**（如对已有行 **`UPDATE`** 递增数量）；否则唯一约束将导致写入失败。**`/wormhole`** 协议 Sheet 内编辑行项目亦同。
        *   **无可用协议（第二区块 Select `items` 为空）：** 用户仍可打开 Sheet（若入口不禁用）。此时 **Sheet 主内容区无协议摘要、无行项目、无总价**（**空壳**），**仅**在可滚动主区**居中或顶对齐**展示与 **Select 空列表** **完全相同**的提示文案——即默认：**「当前校准宇宙下暂无编制中的虫洞协议。请先切换校准宇宙，或前往虫洞协议页新建。」**（导航入口：**Nav · 虫洞倾泻协议** → **`/wormhole`**，见 §3.4）；若实现已判定用户名下**全局**无任何 **`DRAFT`**，则与 Select 一致改为：**「暂无编制中的虫洞协议。」** **顶栏箭头**（收起 / 可选加宽）**保留**，以便用户在不返回第二区块的情况下关闭 Sheet；**「提交签订 / 结账」**应**禁用**或**不渲染**，避免无协议时的伪导航。
        *   **Sheet 结构（自上而下）：**
            1. **顶栏控件（左上）：** 见上条「面板内顶栏」；其下再接摘要与列表（顶栏**不参与**行项目滚动区，或随头部 sticky，由实现择一，须避免遮挡总价条）。
            2. **协议摘要区（文案）：** 展示当前选中虫洞协议的**战术代号**（`protocol_name`）及**一行叙事辅助说明**（例：「以下为该出向编制已挂载的军火；移除行项目相当于从编制单撤下该 SKU。」——可本地化）。因本区块仅可选 **`DRAFT`**，**可不重复强调状态**，必要时用小字标注「编制中」。
            3. **行项目列表（类电商购物车）：** 遍历当前协议之 **`wormhole_items`**（联结 **`weapons`**、**`weapon_merchandising`**）。**每一行**布局为：
                *   **左侧（Card 外）：** **垃圾桶图标按钮**（`Trash2` 等），表示从本协议编制中**移除该条 `wormhole_items` 行**（**不**删除协议本身）。图标与 Card **水平对齐**，与 Card 之间保留固定间距。
                *   **右侧 Card（shadcn `Card`）：** 内含：
                    *   **配图：** **1:1** 武器产品图（持久化字段若暂无，用占位图；后续可增 `weapons` 配图 URL 并与脚手架对齐）。
                    *   **标题：** **`weapon_merchandising.product_title`**（无营销副表时可退化为 SKU 展示名，实现兜底）。
                    *   **单价：** **`DRAFT`** 下展示目录标价 **`weapons.list_price_qc`**；**`price_snapshot_qc` 非空**（`PENDING` / `COMPLETED`）时以快照价为结算展示基准（**§2.4**）。
                    *   **数量：** 对应脚手架 **`wormhole_items.quantity`**（**int**）；若产品后续改为「仅展示不可改」，在本 Sheet 内只读即可。
            4. **总价条（始终可见）：** 在卡片列表**底部**，使用 **`sticky bottom-0`**（推荐：位于 Sheet **内容滚动区**底部，随 Sheet 内滚动但不会滚出视窗；若实现困难可用 **`fixed` 底部栏**，须避开 Sheet 关闭条与安全区）。展示**应付 QC 合计**：**`DRAFT`** 为目录价 × 数量之和；**已锁价**后与 **`wormhole_protocols.total_cost_qc`** 一致（**§2.4**）。文案示例：**「应付合计：〈n〉 QC」**。
            5. **签订（主行动，Sheet 内完成）：** **「提交签订 / 结账」**（文案待定）置于总价条**下方**或与总价条同一 **sticky** 操作区；**不**使用独立路由页面。点击触发 **`DRAFT` → `PENDING`** 业务提交（**§2.4 / §4.3**，扣款、锁价、流水由服务端事务完成）；失败时 Sheet **保持打开**并展示错误；成功后可关闭 Sheet 并刷新 Select / 列表。**无协议**、**空篮**（无可结账行）时该按钮**禁用**或**不渲染**。本 Sheet **仅**绑定 **`DRAFT`** Select，**不涉及** `PENDING` / `COMPLETED` 的 Sheet 分支。

    *   **用户体验文案（建议稿，可本地化）：**
        *   **区块标题：** 「出向编制 · 虫洞协议」或「选择虫洞倾泻协议」。
        *   **辅助说明：** 「选定后将作为当前会话绑定的出向管线。新建或删除整条协议请在虫洞协议页（`/wormhole`）操作。」
        *   **Select 占位符：** 「请选择一条虫洞协议…」
        *   **空列表：** 默认：**「当前校准宇宙下暂无编制中的虫洞协议。请先切换校准宇宙，或前往虫洞协议页新建。」** 若实现检测到用户名下**全局**无任何 **`DRAFT`**（与 Carousel 选项无关），可简化为：**「暂无编制中的虫洞协议。」**
        *   **打开 Sheet 按钮：** 「查看出向清单」「查看协议内武器」（二选一产品化）。
        *   **Sheet 无协议（与 Select 空列表同源）：** 与 **「空列表」** 条**同一套文案**（默认 / 全局无 **`DRAFT`** 两种），见上款「无可用协议」。
        *   **Sheet 空篮：** 「当前协议内尚无挂载武器。」（**适用：** 已有选中 **`DRAFT`** 协议，但该协议下 **`wormhole_items` 为空**；与「第二区块无协议」区分。）
        *   **无障碍：** Select、`Sheet`、垃圾桶按钮、**面板内左右箭头按钮**均保留 **label / aria**（如「收走出向清单」「加宽面板」）；焦点陷阱遵循 shadcn 默认。

*   **第三区块：武器营销展示 — Grid + Card（shadcn）**

    *   **整体：** 主内容区在第二区块**之下**，为**武器目录**的**固定栅格**展示；**不**在区块底部提供 **Pagination**；**不**在 MVP 中采用「滚动触底再拉取 / 虚拟列表动态刷新」等增量加载策略。
    *   **布局骨架：** 使用 **CSS Grid**（或等价的响应式 Grid 系统）排布；**每个栅格 cell 对应一枚** shadcn **`Card`**，一卡一 SKU（与 **公理 III** 及第一区块**校准上下文**下的目录结果一致；**全宇宙聚合**时与「欢迎」Carousel 项规则一致，见上）。
    *   **Card 内信息（MVP 先定版式、不定稿内容）：** 以**营销层**为主（持久化以 **`weapon_merchandising`** 等脚手架为准，见 **§1.4**），具体**文案、徽标、卖点条**等**产品稿待定**；**媒体区宽高比**（如封面图、轮播区）**待定**——本阶段**仅约定**：Card 内具备**可区分的媒体区 + 文本/价签区**的**上下或复合分区**，由设计与实现择一 **layout** 冻结即可，细节不阻塞开发搭架。
    *   **数据加载（MVP）：** 假设**武器总量较少**；进入首页或**校准上下文**变化时，**一次性**请求并渲染**当前过滤条件下**的**全部**武器 **Card**（**全量返回、前端全量挂载**）。**不**要求：分页参数、游标、按视口懒加载、**infinite scroll**。

### 3.4 虫洞倾泻协议页 `/wormhole`

*   **路由与壳层：** 挂载于 **§3.2** 业务壳（Nav + 主内容区）；与 Nav **「虫洞倾泻协议」** 指向的命名空间一致，**建议页面路径**为 **`/wormhole`**（子路由 **`/wormhole/*`** 预留）。

*   **整体：** 主内容区**垂直排列**三个 section（自上而下）。

*   **第一区块：校准宇宙 — Carousel（共享组件）**
    *   与 **§3.3** 首页第一区块**同一套「校准宇宙」React 组件**（Carousel 条目、切换行为、`universes` 数据来源一致），确保终端侧**校准上下文**跨页一致。
    *   **本页列表数据源：** 下述三段协议区仅展示 **`target_universe_id` = 当前 Carousel 选中平行宇宙之 `universes.id`**、且 **`wallet_id` = `auth.uid()`** 之 **`wormhole_protocols`**。**建议：** 本页 Carousel **不包含「欢迎」项**（仅平行宇宙条目），避免「未锚定宇宙」时列表语义歧义；若必须与首页完全同一数据源，则选中「欢迎」时三段行为须**产品定义**（例如置空并提示选择宇宙），**MVP 推荐关闭欢迎项**。

*   **第二区块：控制面板 — 创建虫洞协议表单**
    *   **范围：** **仅创建**，即提交后在库中插入一条 **`wormhole_protocols`**（默认 **`status = 'DRAFT'`**，与 **§2.4** 一致）。**不包含**删除或编辑已有协议——均在**第三区块**点开 **Sheet** 后操作（见下）。
    *   **字段与校验：**
        *   **必填：** **`target_universe_id`**（投送目的地宇宙；**取自当前 Carousel 选中之平行宇宙**，表单内可只读展示或隐藏字段自动带出，**禁止**未选宇宙时提交）、**`protocol_name`**（战术代号）。**同一 `wallet_id` 名下 `protocol_name` 不得重复**（创建前校验；与数据库唯一约束一致，见 **第二级** **`docs/schema/all.dbml`**）。
        *   **必填且默认：** **`coord_x`、`coord_y`、`coord_z`** 三轴坐标均为**必填**，表单控件的**默认值为 `0`**（浮点）；用户可改，未改则写入 **`0.0`**。
        *   **`total_cost_qc`** 与 **`wormhole_items`** 不在此表单创建；创建时写入 **`total_cost_qc = 0`**（脚手架 **default 0**），有行后由**服务端在 `DRAFT` 下**按 **§2.4** 实时重算。
    *   **版式：** 「控制面板」为区块标题下的表单区域（**shadcn `Form` / `Input` / `Button`** 等）；具体排版与错误提示（含「战术代号重复」）后续细化。

*   **第三区块：协议列表 — 按状态分三段（垂直堆叠 + 区内 Card Grid）**
    *   **页内宏观布局（MVP 写死）：** 三个状态区 **仅允许自上而下垂直排列**——顺序固定为 **`DRAFT` 区块 → `PENDING` 区块 → `COMPLETED` 区块**（同宽、占满主内容区宽度，**禁止**三栏并排作为默认实现；小屏下亦不改为横向三列）。各区块有**小节标题** `DRAFT` / `PENDING` / `COMPLETED`；区块**内部**用 **CSS Grid** 排布 **shadcn `Card`**，一 **Card** 对应一条 **`wormhole_protocols`**。文档所称「三列」指**逻辑上的三份 status 列表**，**不是**页面横向三列。
    *   **空状态（该区无任何协议行时，须占位文案，可本地化）：**
        *   **`DRAFT` 区为空：** **「当前校准宇宙下暂无编制中的虫洞协议。」**（可选一行辅助：**「可在上方创建表单新建一条，或切换校准宇宙。」**）
        *   **`PENDING` 区为空：** **「当前校准宇宙下暂无处理中的虫洞协议。」**
        *   **`COMPLETED` 区为空：** **「当前校准宇宙下暂无已完成的虫洞协议。」**
    *   **数据：** 在**当前选中宇宙**约束下，按 **`status`** 分流入三段 Grid；**不**做分页、**不**做滚动懒加载（**MVP** 与 §3.3 武器 Grid 相同假设：**条数少、一次加载**）。**MVP** **不**要求表头筛选、多字段排序等；**顺序按查询默认（如主键/插入顺序）即可**，避免过度设计。
    *   **Card（仅只读展示）：** **不得**在 Card 上放置删除、编辑等**操作控件**；Card **仅用于展示**本条协议的关键信息（具体字段与样式**未定**，至少可辨识 **战术代号**、**status** 等）。**整卡可点击**（或明示「查看 / 管理」之覆盖层，**样式未定**），用于打开下文 **Sheet**。

*   **详情与管理：`Sheet`（共享组件，区别于首页购物篮 Sheet）**
    *   **触发：** 点击某一协议 **Card** 后，自右侧打开 **shadcn `Sheet`**（**`side: 'right'`**，与 §3.3 首页 Sheet 一致）。**所有**对既有协议的**修改、删除、签订等允许操作**均在 **Sheet 内**完成，**不在 Grid Card 上**重复入口。
    *   **共享性：** 将该 **`Sheet` 实现为可复用 React 组件**（例如传入 `wormhole_protocol_id`、当前行数据、**`status`**），**勿与**首页第二区块「购物篮」Sheet **混为同一业务组件**（二者职责不同：本页为**协议壳与生命周期管理**，首页为**行项目购物车**）。
    *   **按 `status` 裁剪功能（须严格遵守）：** **同一组件**须根据 **`wormhole_protocols.status`** **隐藏或禁用**不适用入口，避免 **`PENDING` / `COMPLETED`** 仍出现签订、清除等误导控件。
        *   **`DRAFT`：** 可编辑 **`protocol_name`**（**PATCH** / RPC；服从 **§2.4**）；**「清除协议」**（删除整条协议及 **`wormhole_items`**；**MVP 建议**仅 **`DRAFT` 可删**）；**签订**：提供 **「提交签订 / 结账」**（与 **§3.3** 购物篮同类），**在 Sheet 内**触发 **`DRAFT` → `PENDING`**（**§2.4 / §4.3**），**不**跳转其它路由；失败保留 Sheet 并提示错误。后续若增加坐标、行项目编辑，**默认亦在本 Sheet**。
        *   **`PENDING` / `COMPLETED`：** **不得**展示 **「提交签订」**、**「结账」**、**「清除协议」** 等仅编制 / 签订阶段可用的入口；**`protocol_name`、坐标等须只读展示**（不可表现为可提交表单）。内容以**战术代号、状态、快照总价、坐标与（可选）行项目只读列表**为主。**顶栏箭头 / 关闭**等导航控件可保留。
    *   **顶栏与交互：** 可与 §3.3 类似保留 **左上箭头** 收起；**具体布局未定**，仅约定使用 **shadcn `Sheet` + `Card` 体系**。

## 4. 用户故事（User Stories）

本节以叙事化场景描述终端侧**可验收**行为，与 **第 1 章** 公理、**第 2 章** 持久化及 **第 3 章** UI/UX 约定对齐；实现细节以本文与脚手架为准，冲突时**改实现、不改用户价值表述**。**MVP 范围：** 本文**暂不规定**虫洞协议与黑洞协议的**创建费、手续费**等定价细节（视为后续版本再引入）；冲突时以届时定价规则为准，本章用户故事**不以扣费为前置条件**。

### 4.1 链入后，跨宇宙浏览目录并预编虫洞备货

- **作为** 一名已「亚空间链入」的**跨界特工**，**我希望** 在终端里**切换校准宇宙**（文首术语：**专指终端浏览上下文**，与肉身所在物质宇宙无关——人在宇宙 A，仍可切到 **B / C / D…** 查看对应目录），仅看到**当前浏览上下文**下物理法则**允许击发**的军火 SKU，**以便** 为自己备装、或为**身处其它宇宙的协作方代选**武备，并将意图建立在 **QC 可支付** 与 **公理 III** 目录过滤之上（与 **身份即钱包** 一致）。
- **作为** 同一主体，**我还希望** 在尚未抵达目标宙域前，就能先**创建并留在 `DRAFT`** 一条 **虫洞倾泻协议**（选定与 SKU 原生宇宙一致的 `target_universe_id`、填好坐标等），待本人到达该宇宙后再**提交为即时出运**，**以便** 减少临场编排成本、落地「提前备货、到时执行」的作战节奏。
- **验收要点（摘要）：**
  - 链入成功后存在与 `auth.users.id` 同一 UUID 的 `wallets` 行。
  - 目录结果随终端**当前校准宇宙**（浏览上下文，见文首术语）过滤；不可击发 SKU 不得作为该上下文下的可购/可选结果返回。**例外：** 首页 Carousel 选中 **§3.3「欢迎」项**时为**全宇宙聚合列表**，不适用单一 `native_universe_id` 过滤。
  - **虫洞协议**层面：`wormhole_protocols.target_universe_id` 必须与协议内各 `wormhole_items` 对应武器的 `native_universe_id`**一致**；**不**额外要求该目标等于特工肉身所在宇宙。
  - **DRAFT 边界：** 协议行一旦创建（`DRAFT`），**禁止**修改 `target_universe_id`；若需改投送目的地宇宙，须**新建**一条虫洞协议。`coord_x` / `coord_y` / `coord_z` 等编制字段允许在 `DRAFT` 内继续修改（见 **§2.4**）。

### 4.2 黑洞虹吸：自建协议并入账 QC

- **作为** 一名已链入且持有 **§2.1** 钱包的**跨界特工**，**我希望** 在 **`/black-hole`**（**§3.5**）自行**新建**一条或多条 **`black_hole_protocols`**（**`DRAFT`**），填写**战术代号**与坐标，物质所在宇宙由**校准宇宙 Carousel**锚定之 **`target_universe_id`**（**不**由我填报拟入账 **QC**），**以便** 按入向管线叙事开启虹吸编制。
- **作为** 同一主体，**我希望** 在 **`DRAFT`** 上提交 **「提交虹吸」**（见 **§3.5 Sheet**）后走**即时处理**（**无**延迟执行或预约时间选择，与 **公理 IV · 黑洞协议**、**§2.3** 一致）；在协议进入 **`COMPLETED`** 后看到 **`balance_qc` 增加**与 **`BLACK_HOLE_INTAKE`** 不可变流水，入账数额由 **§2.3** **NSM 侧裁定**（**MVP** 为 **\[1000, 1000000\]** 随机整数占位），**以便** 将远域物资**等价交换**为亚空间内可复用的 **QC**（与 **公理 I** 一致）。
- **验收要点（摘要）：**
  - 首链入成功：**仅要求** `wallets` 行存在（与 **`auth.users.id`** 对齐）；**不要求**同时存在任何 **`black_hole_protocols`** 行——黑洞协议**仅**在用户于 **`/black-hole`** **创建**后出现。
  - 提交后状态机与入账规则仍服从 **§2.3**（仅 `COMPLETED` 时写入 **`credited_amount_qc`** 并更新余额、记账；失败回滚、无失败终态，见 **§2.5**）。
  - 入账 **QC** 须由 **NSM 侧裁定**产生，**禁止**以用户自报金额作为入账依据（与 **§2.3 定价权** 一致）；**MVP** 随机区间见 **§2.3 · MVP · 无明细子表与模拟估值**。
  - **MVP：** 不在用户故事层要求「创建黑洞 / 虫洞须先扣手续费」；若后续版本引入协议费，须单独修订**第 4 章**与 **§2** 约定。

### 4.3 虫洞倾泻：提交即锁价扣款并完成出向

- **作为** 一名已在 **§4.1** 场景下**编制完成**（`DRAFT`）虫洞倾泻协议的**跨界特工**，**我希望** 在确认 `wormhole_items` 与目标宇宙一致、且**钱包 `balance_qc` 足以覆盖**协议总额后，执行**一次提交**使协议进入 **`PENDING`**：系统按 **§2.4** 自目录锁价、扣减 QC，并**立即**进入出向运输处理，**以便** 体现 **公理 IV · 虫洞协议**「**提交即即时运输**」、且全过程可追溯至 **`WORMHOLE_CONSUME`** 流水与 `price_snapshot_qc`（与 **§2.2** 一致）。
- **作为** 同一主体，**我希望** 在 **Supabase 侧整笔业务事务**（锁价、扣款、记流水、协议状态推进）**提交成功**后，协议即进入 **`COMPLETED`**，**以便** 在模拟工程内与 **§2.4**「事务成功即闭环」一致，而**不依赖**真实世界物流回执。
- **验收要点（摘要）：**
  - `DRAFT` → `PENDING`：仅在业务规则满足时发生（含余额充足、**§2.5** 虫洞物理门禁通过）；进入 `PENDING` 时自目录写入各行的 **`price_snapshot_qc`**（此前 **`NULL`**），并写入协议级快照 **`total_cost_qc`**；产生与协议关联的 **`WORMHOLE_CONSUME`**（金额为负，与扣款一致）。
  - `PENDING` → `COMPLETED`：在与上述扣款**同一成功提交**的数据库事务内（或紧接的同一业务原子流程中）将 `wormhole_protocols.status` 置为 **`COMPLETED`**（**§2.4**）；**不**使用 FAILED 终态枚举（**§2.5**）。叙事上若再描述「虫洞运输事故」，**不**回滚已扣 **QC**（**§2.4 类自然灾害**）。
  - **无预约列：** 持久化不包含「计划执行时间」类字段（见 **§2.4、公理 IV**）。

### 4.4 触碰到门禁时：失败可感知、数据不留「失败协议」

- **前提（为何门禁必须在 `DRAFT` → `PENDING` 卡死）：** 虫洞管线在 **§2.4** 中为**模拟暴力闭环**——**`PENDING` 在当前流程下一定会走到 `COMPLETED`**（同一 Supabase 事务内锁价、扣款、落账、置终态）。叙事上符合 **NSM 亚空间黑市**「一旦出货管线点火即交割」的冷酷性；因此**绝不能**把本该拦在门外的错误放进 `PENDING`**。**买入意愿与卖出接单在越过门禁的瞬间即视为不可逆承诺**（此后仅余 **§2.4** 叙事层不可抗力，不退款）。
- **作为** 一名操作终端的**跨界特工**，**我希望** 在**余额不足以支付**拟提交虫洞协议、或**黑洞 / 虫洞协议**因坐标无效、宇宙 / SKU 不一致、目录物理不可行等原因**无法合法进入处理**时，得到**明确错误反馈**，且数据库中**不出现** status 为 FAILED 的协议行，**以便** 符合 **§2.5（景 A）**：**事务未提交**时整笔回滚，不把失败写成协议终态。
- **作为** 同一主体，**我还希望** 在虫洞场景下，若因门禁失败而**从未进入 `PENDING`**，则**不发生**扣款与 **`WORMHOLE_CONSUME`** 记账，**以便** **§2.2** 账本仅记录真实发生的价值变动。
- **`DRAFT` → `PENDING`（虫洞）必要条件（验收）：**
  - **余额充足（第一要素）：** 须由 **Supabase 服务端**在提交事务中检验（例：扣款前 `balance_qc` ≥ 应付总额）；不满足则**拒绝进入 `PENDING`**、**不扣款**。
  - **宇宙门禁：** `wormhole_protocols.target_universe_id` 须与**各行** `wormhole_items` 所引用 `weapons.native_universe_id`**一致**。**默认由前端**在提交前完成核对（黑市终端叙事：**买家自负对齐协议与货品宇宙**）；若 **Supabase** 端能以触发器、RPC、约束等**低成本**实现同一规则，**建议叠加**服务端校验，避免伪造请求篡改明细。
  - **坐标、物理可行性及其它虫洞门禁**（公理 III 可击发等）仍按 **§2.5**；不满足则停留 **`DRAFT`** 或拒绝提交。
- **验收要点（摘要）：**
  - **`black_hole_protocols` / `wormhole_protocols` 的 status 枚举**仍仅 **`DRAFT` / `PENDING` / `COMPLETED`**；不因业务失败新增 ABORTED / FAILED 等终态（见 **公理 IV、§2.5**）。
  - **景 A（事务未提交）：** 门禁失败、余额不足等——**回滚**或阻止提交；用户可见错误信息（实现自定文案与码表）。
  - **虫洞 · 景 A：** 未成功进入 `PENDING` 时，**无**对应扣款流水。
  - **虫洞 · 景 B（事务已提交）：** 价金已扣且协议已 **`COMPLETED`**（**§2.4**）后，叙事上「运输失败」**不退款**、**不冲正流水**，归入 **§2.4** 不可抗力；**不**要求实现层再模拟 FAILED 行。
